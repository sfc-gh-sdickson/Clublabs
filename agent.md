## Use Case Definition

```Clublabs Insights - Technical Overview & Snowflake Integration
Project Vision & Goal The Insights project is an internal analytics platform designed to provide deep, actionable insights into the performance and user behavior of Clublabs’s various apps, starting with AAA's Emergency Roadside Service (ERS) chatbot and Online Bind web app. The primary goal is to move from reactive problem-solving to proactive, data-driven optimization of the user journey, thereby improving member satisfaction and service efficiency. This platform will serve as the single aggregation point of Clublabs apps’ operational and performance logs, feeding everything from executive dashboards to an AI-powered agent that provides daily operational digests, conversation analytics interface, and visualization AI- generated charts.
Current Challenges: Our current solution relies on: fragile NLP-to-SQL chatbot interface, lack of low latency, historical context management, manual data acquisition via scripts and ad hoc data source integration. This process is manual, unreliable, and does not provide the real-time, granular data needed to effectively manage and improve the service. This has created a significant capabilities gap that we plan to close with a modern, scalable data architecture.
Proposed Snowflake Architecture: A Scalable ELT Pipeline We are architecting a new, streaming-first pipeline that leverages the core strengths of AWS and Snowflake. The architecture is designed for scalability, maintainability, and near real-time data availability. Data Flow: AWS S3 (us-east-1, us-west-2) → Snowpipe (Auto-Ingest) → Raw Staging Table (VARIANT) → Streams & Tasks → Structured Analytics Tables A few separate data flows to ingest production conversation transcripts and IVR data from non- AWS sources, such as MongoDB and AAA internal systems, might need to be created with help from Snowflake support team. Key Implementation Details: • Source Data for AWS: Application logs are continuously generated by the ERS chatbot, OLB app and backed up from AWS Kinesis to S3 buckets across two regions (us-east-1 and us-west-2) as compressed JSON files. • Ingestion (Load): We will use Snowpipe to automatically and continuously ingest these raw,
compressed JSON logs as they land in S3. Two pipes, one for each region, will feed a single raw staging table, seamlessly merging the data. • Transformation (Transform): Raw data will be loaded into a VARIANT column. A Snowflake Stream will track new records, and a recurring Task will execute a MERGE statement to:
Parse the semi-structured JSON.
Filter for only the relevant ERS chatbot application logs.
Transform and load the clean data into a structured, star-schema model suitable for BI and analytics.
• Analysis & AI/ML: Once the data is structured, we plan to leverage Snowflake's advanced capabilities:  • Business Intelligence: Standard SQL queries will feed our BI dashboards for monitoring KPIs. • AI/ML: We will heavily utilize Snowpark and Snowflake Cortex for sophisticated analysis, including user feedback sentiment analysis, anomaly detection, and natural language queries against our data. 4. Key Use Cases & Desired Outcomes • Real-time Funnel Analysis: Track user progression through the 19-step chatbot flow to identify drop-off points. The NLP-to-SQL will be exposed to Insights v1.0 app to allow user entering custom questions. The answers produced by Snowflake will also be used by an AI agent to generate visualization. • Sentiment Analysis: Analyze user feedback at scale to quantify member satisfaction and identify emerging issues. • Anomaly Detection: Automatically flag unusual patterns in chatbot errors or user behavior. • AI-Powered Summarization: The final structured data will be consumed by an internal AI agent (insights-bot) to generate daily performance digests for the project team.
```

The implementation is structured into four main phases, moving from initial setup to advanced analytics and final solution deployment.Phase 1: Foundation & Initial Ingestion (S3 Data)

Goal: Establish the core infrastructure to continuously load raw log data from AWS S3 into Snowflake.
Setup AWS Infrastructure:
Provision and secure S3 buckets in both us-east-1 and us-west-2 regions.
Configure ERS Chatbot and OLB applications to continuously back up compressed JSON logs from AWS Kinesis to the respective S3 buckets.
Snowflake Setup:
Create a single Raw Staging Table with a dedicated VARIANT column to hold the raw JSON.
Configure two separate Snowpipe (Auto-Ingest) instances. Each pipe will monitor one of the S3 regions (us-east-1 and us-west-2) and feed the data seamlessly into the single Raw Staging Table.
Phase 2: Data Transformation and Modeling

Goal: Create a reliable, scheduled process to clean, structure, and model the raw data for analysis.
Data Modeling:
Design the optimal Star-Schema Model for the Structured Analytics Tables, focusing on key entities (e.g., user journey steps, service requests, conversation metrics).
Transformation Logic:
Create a Snowflake Stream on the Raw Staging Table to track only new records since the last processing run.
Develop a Snowflake Task to run on a recurring schedule (e.g., every 5 minutes). This task will execute a MERGE statement that:
Reads the new data from the Stream.
Applies JSON parsing to extract all necessary fields from the VARIANT column.
Filters for only the relevant ERS chatbot application logs (and potentially other high-priority data).
Transforms and standardizes the clean data.
Loads the structured data into the Star-Schema tables.
Phase 3: Auxiliary Data Integration

Goal: Integrate non-AWS sources for a complete view of the operational data.
External Data Ingestion Flows:
Consultation: Engage the Snowflake Support Team to architect custom ingestion solutions for production conversation transcripts and IVR data residing in MongoDB and AAA internal systems.
Development: Implement the new data flows (e.g., using external staging, API connectors, or other Snowflake-recommended patterns).
Integration: Map and integrate this auxiliary data into the existing Structured Analytics Tables to enrich the core dataset.
Phase 4: Advanced Analysis & Outcome Activation

Goal: Leverage Snowflake's advanced capabilities to achieve the desired outcomes and close the capabilities gap.
Business Intelligence (BI):
Connect BI tools to the Structured Analytics Tables.
Develop executive and operational dashboards for monitoring key performance indicators (KPIs) and service health.
Create a real-time funnel analysis dashboard to track the 19-step chatbot flow and identify drop-off points.
AI/ML & Advanced Capabilities:

Sentiment Analysis: Analyze user feedback at scale to quantify member satisfaction and detect emerging issues.

AI-Powered Summarization:
Integrate the internal AI agent (insights-bot) to consume the final structured data and automatically generate daily performance digests for the project team.

### SQL UDF Return Types (VERIFIED)

| Function Returns | Correct Return Type |
|------------------|---------------------|
| `ARRAY_AGG(...)` | `RETURNS ARRAY` |
| `OBJECT_CONSTRUCT(...)` | `RETURNS OBJECT` |
| Single scalar value | `RETURNS VARCHAR/NUMBER/etc` |

**DO NOT USE:**
- `RETURNS VARIANT` for `ARRAY_AGG` or `OBJECT_CONSTRUCT`
- `LANGUAGE SQL` clause in SQL UDFs

### SQL UDF Syntax (VERIFIED)

```sql
-- Correct syntax for scalar UDF returning ARRAY
CREATE OR REPLACE FUNCTION AGENT_GET_DATA()
RETURNS ARRAY
AS
$$
SELECT ARRAY_AGG(OBJECT_CONSTRUCT(
    'key1', COLUMN1,
    'key2', COLUMN2
)) FROM (SELECT * FROM TABLE LIMIT 50)
$$;

-- Correct syntax for scalar UDF returning OBJECT
CREATE OR REPLACE FUNCTION AGENT_GET_SUMMARY()
RETURNS OBJECT
AS
$$
SELECT OBJECT_CONSTRUCT(
    'metric1', (SELECT COUNT(*) FROM TABLE1),
    'metric2', (SELECT AVG(COLUMN) FROM TABLE2)
)
$$;
```

## Lessons Learned (CRITICAL)

### 1. ALWAYS VERIFY SNOWFLAKE SYNTAX BEFORE WRITING CODE

**What went wrong:** Multiple syntax errors because I guessed at syntax instead of verifying against Snowflake documentation.

**Correct approach:**
- Use `snowflake_product_docs` tool to look up syntax BEFORE writing any SQL
- Use `system_instructions` tool for Cortex Agent, Analyst, and other Snowflake products
- Reference working examples (like Troon Intelligence Agent)

**Specific errors made:**
- Used `RETURNS VARIANT` instead of `RETURNS ARRAY` for `ARRAY_AGG`
- Used `RETURNS VARIANT` instead of `RETURNS OBJECT` for `OBJECT_CONSTRUCT`
- Used `LANGUAGE SQL` clause which is invalid for SQL UDFs
- Used `type: "procedure"` instead of `type: "function"` for agent tools
- Used `search_service:` instead of `name:` for Cortex Search resources
- Used JSON format instead of YAML for agent specification

### 2. COMPLETE ALL FILES BEFORE STOPPING

**What went wrong:** Generated partial files and stopped without completing the project, leaving merge conflicts and incomplete code.

**Correct approach:**
- Review ALL files in the project at the start
- Create a TODO list for every file that needs to be created/modified
- Do not mark a task complete until the file is verified to compile/run
- Verify file completeness before moving to the next task

### 3. NEVER GUESS - ASK OR RESEARCH

**What went wrong:** Made assumptions about:
- Agent YAML syntax
- SQL UDF return types
- Function naming conventions
- Tool resource configuration

**Correct approach:**
- If unsure about syntax, use documentation tools first
- If documentation is unclear, ask the user for clarification
- Reference working examples from similar projects
- Test small pieces of code before combining them

### 4. ASK QUESTIONS WHEN UNCLEAR

**What went wrong:** Proceeded with assumptions instead of asking for clarification on requirements.

**Questions to ask upfront:**
- What business domain/industry is this for?
- What specific ML models or predictions are needed?
- What data sources exist or need to be created?
- What sample questions should the agent answer?
- Are there any existing working examples to reference?

### 5. VERIFY GIT MERGE CONFLICTS

**What went wrong:** Left merge conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`) in SQL files.

**Correct approach:**
- After any file operations, verify no merge conflicts exist
- Search for conflict markers before marking files complete
- Test SQL files compile before considering them done

### Database Setup (01_database_and_schema.sql)

```sql
CREATE DATABASE IF NOT EXISTS {DATABASE_NAME};
USE DATABASE {DATABASE_NAME};

CREATE SCHEMA IF NOT EXISTS RAW;
CREATE SCHEMA IF NOT EXISTS ANALYTICS;

CREATE OR REPLACE WAREHOUSE {WAREHOUSE_NAME} WITH
    WAREHOUSE_SIZE = 'X-SMALL'
    AUTO_SUSPEND = 300
    AUTO_RESUME = TRUE
    INITIALLY_SUSPENDED = TRUE
    COMMENT = 'Warehouse for {CUSTOMER_NAME} Intelligence Agent';

USE WAREHOUSE {WAREHOUSE_NAME};
```

### Cortex Search Service

```sql
CREATE OR REPLACE CORTEX SEARCH SERVICE {SEARCH_SERVICE_NAME}
  ON {text_column}
  ATTRIBUTES {attr1}, {attr2}, {attr3}
  WAREHOUSE = {WAREHOUSE_NAME}
  TARGET_LAG = '1 hour'
  COMMENT = 'Description of search service'
AS
  SELECT
    {columns}
  FROM {TABLE};
```

### Semantic View

```sql
CREATE OR REPLACE SEMANTIC VIEW {SEMANTIC_VIEW_NAME}
  COMMENT = 'Semantic view description'
AS
  SELECT
    {table}.{column} AS {alias}
      WITH SYNONYMS ('{synonym1}', '{synonym2}')
      COMMENT = '{Column description}',
    ...
  FROM {database}.{schema}.{table}
  ...;
```

## Checklist for New Projects

### Before Starting
- [ ] Confirm customer name and business domain
- [ ] Identify data sources (existing tables or need synthetic data)
- [ ] Determine ML models needed (LTV, churn, risk, etc.)
- [ ] Collect sample questions the agent should answer
- [ ] Get working example project for reference

### During Development
- [ ] Verify ALL SQL syntax against Snowflake docs before writing
- [ ] Test each SQL file compiles before moving to next
- [ ] Check for merge conflicts after any file operations
- [ ] Complete TODO list for every component

### Before Delivery
- [ ] Run all SQL files in order (01-08)
- [ ] Test agent creation succeeds
- [ ] Verify agent responds to sample questions
- [ ] Update documentation with customer-specific details
- [ ] Remove any placeholder values
